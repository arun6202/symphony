Two Valid Entry Points

Brownfield teams often start from database truth. Greenfield teams often start from F# domain types. The platform should support both and converge them into one projection spec.

Database-First

Harvest tables, constraints, comments, keys, and type fidelity from existing systems. Generate a source catalog, then enrich it into governed projections.

Functional DDD

Start with bounded contexts, records, discriminated unions, constrained primitives, and workflows. Generate read models and search projections from domain intent.

New MVP Direction

Unify the codebase around a SQLite miniature first, then repeat the exact pipeline through Oracle end-to-end.

SQLite miniature: schema -> SourceCatalog -> ProjectionSpec -> TSV -> DuckDB -> ETL -> Elasticsearch Oracle production path: dictionary -> SCN extract -> typed TSV -> DuckDB -> ETL -> Elasticsearch -> run ledger Greenfield: F# DDD -> DomainIR -> ProjectionSpec -> same outputs Polyglot: DomainIR -> TypeScript / JavaScript / Kotlin / Swift models + validators

License And Core Language Posture

Apache 2.0 plus MIT dependencies is enterprise-friendly. The architectural answer is permissive/open, but the compiler core should stay uncompromisingly F#.

License Advantage

Apache 2.0 at the root and MIT for embedded SqlHydra make this safe to inspect, fork, adopt, commercialize, and extend with clear attribution rules.

F# Core Rule

Do not weaken the core for C# compatibility. Generate C# OneOf/sealed-record adapters from DomainIR at the boundary.

How This Fares

The shape is strong because it connects categories that are usually separate: schema intelligence, functional modeling, projection, search, catalog, quality, and documentation.

Strong Wedge

Turn existing relational systems into governed Elasticsearch surfaces with mappings, lineage, contracts, tests, and docs.

Main Blind Spot

CDC can move events, but denormalized document semantics decide correctness. Parent changes, deletes, and history policy need product decisions.

SOTA Upgrade

Attach evidence to every claim: spec hash, mapping hash, validation run, lineage coverage, quality result, and run ledger.
