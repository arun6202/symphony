Production backbone

Oracle to TSV to DuckDB to ETL to Elasticsearch.

This path creates a durable, inspectable boundary between source extraction and projection. TSV is simple, DuckDB is the staging/query engine, and Symphony owns the projection semantics.

Why TSV first?

It is streamable, human-inspectable, diffable, easy for DuckDB to load, and perfect for replay/debug before Parquet or direct streaming.

1
Oracle harvest
Read columns, constraints, comments, keys, indexes, and type fidelity into SourceCatalog.
2
SCN extract
Extract source rows with source identity, SCN, row counts, checksums, and query hash.
3
Typed TSV
Emit data TSV, schema JSON, manifest, checksums, and extract log.
4
DuckDB stage
Load TSV, run staging SQL, inspect counts, and support reconciliation.
5
ETL projection
Parse, refine, build documents, assign deterministic IDs, stream to bulk writer.
6
Elasticsearch validation
Create validation index, bulk load, smoke query, strict mapping rejection, quality suite, alias plan.
Evidence chain: source catalog hash + projection spec hash + TSV manifest hash + mapping hash + bulk manifest hash + validation result + alias result = run ledger
