Fresh architecture take

Support brownfield database-first and greenfield functional DDD.

Symphony should not force one religion. Existing enterprises start from database truth. New F# systems can start from domain language, discriminated unions, records, and workflows. Both paths should converge into a typed projection spec.

The unifying idea

Brownfield starts from what exists. Greenfield starts from what should exist. Both should produce governed projections, executable evidence, and readable handoff docs.

Database-First Development

Start with Oracle, SQL Server, SQLite, or another existing database. Harvest tables, columns, constraints, comments, indexes, and relationships. Then generate typed source models and projection specs.

Oracle dictionary SqlHydra Constraints Lineage

Functional DDD

Start with domain concepts, constrained primitives, records, discriminated unions, commands, events, and workflows. Persistence and search are boundary projections, not the domain core.

What This Changes

The architecture should expose two ingress paths instead of assuming every model starts in the same place.

A
Database-first ingress
Oracle harvester to SourceCatalog to generated F# source types to draft projection spec to human enrichment.
B
Domain-first ingress
F# bounded context to domain events and read-model declarations to projection spec to generated persistence/search adapters.
C
Shared compiler
Once a ProjectionSpec exists, the same folds produce mappings, bulk artifacts, catalog output, tests, contracts, and docs.
